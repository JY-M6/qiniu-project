## 1. 用户故事 (User Stories) 规划与实现

我们在立项之初规划了多维度的用户故事（User Stories），并最终在 MVP 版本中高标准地实现了绝大部分核心场景：

### 规划与实现对照表

| 序号 | 规划的用户故事 (Planned User Story) | 最终实现状态 (Implemented Status) | 实现细节 / 替代方案 |
| :--- | :--- | :--- | :--- |
| **1** | **视频流画面传输**：用户希望开启摄像头，让 AI 实时看到视频里的人和环境。 | **✅ 已实现 (采样优化版)** | 浏览器调取 WebRTC 视频流，在本地 Canvas 以 **1s 间隔**抽取视频帧并进行 320x240 压缩发送，替代昂贵的连续视频流推送，确保视觉识别精准。 |
| **2** | **无感语音对话**：用户希望打开麦克风直接用中文说话，AI 听到后自动应答，免去打字烦恼。 | **✅ 已实现 (边缘端处理)** | 基于 HTML5 **Web Speech API** 在浏览器边缘端直接将语音识别为文字，通过 WebSocket 推送至后端，避免了高昂的云端 STT 费用。 |

## 2. 端云协同成本控制策略 (Cost Control)

在开发与商业化运营大模型应用时，**云端 API 费用**和**带宽流量成本**是应用能否可持续生存的关键。我们在 AURA VISION 中设计了多项独创的“端云协同”成本控制技巧：

### 💡 成本控制技巧对比分析

#### 技巧一：视频流传输控制 (Video Stream vs. Frame Sampling)
*   **想法/方案 A (昂贵)**：直接将 WebRTC 原始视频流（如 1080p RTMP/WebRTC）持续推送到云端服务器，并在云端提取帧进行识别。
    *   *缺点*：造成极高的云端服务器 CPU/GPU 编解码负载、恐怖的流量带宽开销，且大模型无法以 30fps 处理视频，会产生极高的话费浪费。
*   **实际采用方案 B (高效省钱) ⭐**：**边缘端帧采样与压缩**。
    *   我们在浏览器前端（边缘端）通过 Canvas 每隔 1 秒提取一次当前摄像头的视频帧。
    *   在提取时，直接将图片的分辨率裁剪并压缩至 **320x240** 像素，质量因子设为 **0.7**（JPEG 压缩率）。
    *   **成效**：使得每张图像的大小控制在 **5KB - 15KB** 左右，相比直接传输视频流，**节省了 98% 以上的带宽流量**，且完全符合 `qwen-vl-max` 等视觉大模型的图像理解尺寸要求，云端处理成本骤降。

#### 技巧二：语音识别 (Cloud STT vs. Edge STT)
*   **想法/方案 A (昂贵)**：将用户的麦克风音频流录制并实时上传到云端，调用阿里的语音识别服务或 OpenAI Whisper 接口。
    *   *缺点*：每分钟语音识别都需要付费，在大流量高并发下会产生无法估量的 API 账单。
*   **实际采用方案 B (零成本) ⭐**：**浏览器原生 Web Speech API**。
    *   我们将语音识别（STT）逻辑**完全下放到用户的浏览器（边缘端）**去执行。利用用户本机的系统硬件级语音转文字能力，转译为文本后再通过 WebSocket 极速传给后端。
    *   **成效**：**云端 STT 成本降为 0 元**，且识别速度接近即时，大大降低了对话延迟，体验非常自然。

#### 技巧三：上下文历史滑动窗口 (Sliding Window & Non-duplication)
*   **想法/方案 A (昂贵)**：将整场会话的所有对话记录（包含每次上传的 Base64 图像数据）全量保存并累加，每次对话都将历史所有的图片和文本全部打包发给大模型。
    *   *缺点*：多模态图像的 Token 数量极大（单张图可达数百 Token），历史记录累加会导致每次调用的 Input Token 呈指数级上涨，产生巨额账单并极易超出大模型上下文限制。
*   **实际采用方案 B (智能优化) ⭐**：**滑动窗口与图像单次过滤**。
    *   后端 [SessionContext.java](file:///d:/Java/antigravity/人工智能视觉对话助手MVP/backend/src/main/java/com/antigravity/aivision/model/SessionContext.java) 实现了 **最近 5 轮** 对话的滑动窗口，超出 5 轮的历史对话自动剔除，控制 Token 上限。
    *   历史记录（History）在构建大模型 Payload 时，**仅发送对话文本，不重复发送历史图片**，图片仅在用户发送的“当前轮次（Latest Turn）”中单次携带。
    *   **排重校验**：在后端增加了排除遍历最后一项的逻辑，防止因前端先加入历史导致的单次请求中发送重复的最新内容。
    *   **成效**：每次对话请求的 Input Token 数控制在极低水平（约数百 Token），相比全历史图片上传，**Token 消耗减少了 90% 以上**。

#### 技巧四：多级模型路由与阶梯计费 (Model Routing)
*   **实际采用方案 ⭐**：
    *   我们在快速预设中提供了多级模型选项：例如性能顶尖但资费稍高的 **`qwen-vl-max`**（用于需要深度分析的视觉场景）和极其廉价、高性价比的 **`qwen-vl-plus`** / **`gpt-4o-mini`**（用于轻量级语音唠嗑）。
    *   用户可以根据业务预算与精确度要求，在页面上一键无缝切换模型，从根本上实现了运营成本的动态调控。
